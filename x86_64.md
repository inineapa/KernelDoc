## Before You Begin

This document is not an x86 assembly tutorial. It assumes you can read basic assembly — you know what `mov`, `add`, `push`, `pop`, `call`, and `ret` do. What you may not know is *why* the kernel writes `mov %%cr3, %0` in inline assembly, what SMEP and SMAP protect against, how the CPU builds a page table walk from a virtual address, or why `LOCK_PREFIX cmpxchg` appears everywhere in the atomic operations.

This document bridges the gap between "I can read assembly" and "I understand what the x86-specific kernel code is doing and why." We cover the register model, calling convention, control registers, MSRs, paging, segmentation, the IDT, inline assembly patterns, CPU feature detection, and the cache/TLB subsystem — all with references to the Linux 6.19 source.

## 1. x86_64 Registers

### 1.1 General-Purpose Registers

x86_64 has 16 64-bit general-purpose registers:

| Register | 32-bit | 16-bit | 8-bit | Purpose in kernel |
|----------|--------|--------|-------|-------------------|
| `rax` | `eax` | `ax` | `al` | Return value, syscall number |
| `rbx` | `ebx` | `bx` | `bl` | Callee-saved |
| `rcx` | `ecx` | `cx` | `cl` | 4th arg (C) / return addr (SYSCALL) |
| `rdx` | `edx` | `dx` | `dl` | 3rd argument |
| `rsi` | `esi` | `si` | `sil` | 2nd argument |
| `rdi` | `edi` | `di` | `dil` | 1st argument |
| `rbp` | `ebp` | `bp` | `bpl` | Frame pointer (callee-saved) |
| `rsp` | `esp` | `sp` | `spl` | Stack pointer |
| `r8` | `r8d` | `r8w` | `r8b` | 5th argument |
| `r9` | `r9d` | `r9w` | `r9b` | 6th argument |
| `r10` | `r10d` | `r10w` | `r10b` | 4th arg (SYSCALL) / scratch |
| `r11` | `r11d` | `r11w` | `r11b` | Scratch / saved RFLAGS (SYSCALL) |
| `r12` | `r12d` | `r12w` | `r12b` | Callee-saved |
| `r13` | `r13d` | `r13w` | `r13b` | Callee-saved |
| `r14` | `r14d` | `r14w` | `r14b` | Callee-saved |
| `r15` | `r15d` | `r15w` | `r15b` | Callee-saved |

Plus the instruction pointer `rip` and flags register `rflags`.

### 1.2 The System V AMD64 Calling Convention

The kernel (and all of Linux user space) uses the System V AMD64 ABI:

**Function arguments**: `rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9` (in that order). Additional arguments go on the stack.

**Return value**: `rax` (and `rdx` for 128-bit returns).

**Caller-saved** (scratch): `rax`, `rcx`, `rdx`, `rsi`, `rdi`, `r8`–`r11`. The called function may destroy these.

**Callee-saved** (preserved): `rbx`, `rbp`, `r12`–`r15`. The called function must restore these before returning.

**Stack**: Must be 16-byte aligned before a `call` instruction.

This is why, when you look at kernel C code, the first argument to a function ends up in `rdi` in the generated assembly — the compiler follows this convention automatically.

### 1.3 SYSCALL Convention — Different from C

The `syscall` instruction uses a slightly different register mapping:

| Register | Role |
|----------|------|
| `rax` | Syscall number (input) / return value (output) |
| `rdi` | Arg 0 |
| `rsi` | Arg 1 |
| `rdx` | Arg 2 |
| `r10` | Arg 3 (NOT `rcx` — the CPU clobbers `rcx`) |
| `r8` | Arg 4 |
| `r9` | Arg 5 |
| `rcx` | Return address (set by CPU, replaces user's `rcx`) |
| `r11` | Saved RFLAGS (set by CPU, replaces user's `r11`) |

The CPU *clobbers* `rcx` and `r11` during `syscall` — it uses them to save the return address and flags. That is why the 4th syscall argument is passed in `r10` instead of `rcx`. The kernel entry code (`arch/x86/entry/entry_64.S`) must shuffle `r10` → `rcx` before calling the C handler.

## 2. struct pt_regs — The Register Snapshot

When the CPU enters the kernel (via syscall, interrupt, or exception), the kernel saves all registers into `struct pt_regs` (`arch/x86/include/asm/ptrace.h:103–170`). This structure defines the exact stack layout:

```c
struct pt_regs {
    /* Callee-saved registers (pushed by PUSH_AND_CLEAR_REGS in entry_64.S) */
    unsigned long r15;
    unsigned long r14;
    unsigned long r13;
    unsigned long r12;
    unsigned long bp;            // rbp
    unsigned long bx;            // rbx

    /* Caller-saved registers */
    unsigned long r11;
    unsigned long r10;
    unsigned long r9;
    unsigned long r8;
    unsigned long ax;            // rax
    unsigned long cx;            // rcx
    unsigned long dx;            // rdx
    unsigned long si;            // rsi
    unsigned long di;            // rdi

    /* Special fields */
    unsigned long orig_ax;       // syscall number, error code, or IRQ vector

    /* Pushed by CPU on exception/interrupt entry */
    unsigned long ip;            // rip — where we came from
    u16           cs;            // code segment selector
    unsigned long flags;         // rflags
    unsigned long sp;            // rsp — user stack pointer
    u16           ss;            // stack segment selector
};
```

The bottom five fields (`ip` through `ss`) are pushed by the CPU hardware itself when an interrupt or exception occurs. The rest are pushed by the kernel's entry code.

For syscalls, the kernel constructs an equivalent frame manually in `entry_64.S` (lines 101–108):

```asm
pushq   $__USER_DS              /* pt_regs->ss */
pushq   PER_CPU_VAR(...)        /* pt_regs->sp (from TSS) */
pushq   %r11                    /* pt_regs->flags (saved by SYSCALL) */
pushq   $__USER_CS              /* pt_regs->cs */
pushq   %rcx                    /* pt_regs->ip (return addr, saved by SYSCALL) */
pushq   %rax                    /* pt_regs->orig_ax (syscall number) */
```

The `pt_regs` struct is used everywhere: signal delivery reads it to get the return address, `ptrace` uses it to modify registers for debugging, and the syscall return path restores from it.

## 3. Control Registers

x86_64 has several control registers that govern CPU behavior. The kernel accesses them via inline assembly in `arch/x86/include/asm/special_insns.h`.

### 3.1 CR0 — System Control

```c
static inline unsigned long native_read_cr0(void)     // special_insns.h:15
{
    unsigned long val;
    asm volatile("mov %%cr0, %0" : "=r" (val));
    return val;
}
```

Key bits:

| Bit | Name | Purpose |
|-----|------|---------|
| 0 | PE | **Protected Mode Enable** — must be 1 in long mode |
| 1 | MP | **Monitor Coprocessor** — controls FPU/SSE behavior |
| 16 | WP | **Write Protect** — when set, supervisor cannot write to read-only user pages. Critical for copy-on-write to work. |
| 31 | PG | **Paging Enable** — must be 1 for virtual memory |

### 3.2 CR2 — Page Fault Address

```c
static inline unsigned long native_read_cr2(void)     // special_insns.h:33
{
    unsigned long val;
    asm volatile("mov %%cr2, %0" : "=r" (val));
    return val;
}
```

When a page fault occurs, the CPU automatically stores the faulting virtual address in CR2. The page fault handler reads it with `read_cr2()` to know which address caused the fault. This is the address you see in kernel oops messages: "BUG: unable to handle page fault at 0xdead0000".

### 3.3 CR3 — Page Table Base

```c
static inline unsigned long __native_read_cr3(void)   // special_insns.h:44
{
    unsigned long val;
    asm volatile("mov %%cr3, %0" : "=r" (val));
    return val;
}
```

CR3 holds the physical address of the top-level page table (PGD). When the kernel switches processes, it writes the new process's page table address to CR3 — this is the core of the context switch for memory.

**PCID support**: If CR4.PCIDE is set, bits [11:0] of CR3 hold a Process Context Identifier (PCID) — an ASID that tags TLB entries so that a CR3 switch does not flush the entire TLB. Bit 63 is the NOFLUSH bit — when set, loading CR3 does not flush TLB entries with the new PCID.

### 3.4 CR4 — Extended Feature Control

Key bits:

| Bit | Name | Purpose |
|-----|------|---------|
| 5 | PAE | **Physical Address Extension** — required for long mode, enables 4-level paging |
| 7 | PGE | **Page Global Enable** — TLB entries marked global survive CR3 switches |
| 17 | PCIDE | **PCID Enable** — enables process context identifiers in CR3 |
| 20 | SMEP | **Supervisor Mode Execution Prevention** — the kernel cannot execute user-space code. If an exploit redirects kernel RIP to a user-mapped page, the CPU faults instead of executing it. |
| 21 | SMAP | **Supervisor Mode Access Prevention** — the kernel cannot read/write user-space memory directly. The kernel must use `stac`/`clac` (set/clear AC flag) to temporarily allow access — this is what `copy_from_user()` does internally. |

SMEP and SMAP are critical security features. Before SMEP, a common exploit technique was to map shellcode in user space and trick the kernel into jumping to it. SMEP makes that a CPU fault.

## 4. Model-Specific Registers (MSRs)

MSRs are CPU-specific registers accessed via the `rdmsr` and `wrmsr` instructions. The kernel defines their addresses in `arch/x86/include/asm/msr-index.h`.

### 4.1 Key MSRs for System Calls

| MSR | Address | Purpose |
|-----|---------|---------|
| `MSR_LSTAR` | `0xc0000082` | 64-bit SYSCALL entry point — the kernel writes `entry_SYSCALL_64` here at boot |
| `MSR_STAR` | `0xc0000081` | Segment selectors for SYSCALL/SYSRET (kernel CS/SS in bits [47:32], user CS/SS in bits [63:48]) |
| `MSR_SYSCALL_MASK` | `0xc0000084` | RFLAGS mask — bits set here are cleared on SYSCALL entry (disables interrupts) |
| `MSR_KERNEL_GS_BASE` | `0xc0000102` | Shadow GS base — swapped with GS base by `swapgs` instruction |
| `MSR_GS_BASE` | `0xc0000101` | Current GS base address |
| `MSR_FS_BASE` | `0xc0000100` | Current FS base address (used for thread-local storage in user space) |

### 4.2 EFER — Extended Feature Enable Register

`MSR_EFER` (`0xc0000080`) controls fundamental CPU modes (`msr-index.h:20–39`):

| Bit | Name | Purpose |
|-----|------|---------|
| 0 | SCE | **SYSCALL Enable** — enables the `syscall`/`sysret` instructions |
| 8 | LME | **Long Mode Enable** — enables 64-bit mode |
| 10 | LMA | **Long Mode Active** — read-only, indicates 64-bit mode is active |
| 11 | NXE | **No-Execute Enable** — enables the NX bit in page table entries |

### 4.3 Reading and Writing MSRs

```c
// Read an MSR:
u64 val;
rdmsrl(MSR_LSTAR, val);

// Write an MSR:
wrmsrl(MSR_LSTAR, (unsigned long)entry_SYSCALL_64);
```

Internally, these expand to:

```c
asm volatile("rdmsr" : "=a" (low), "=d" (high) : "c" (msr));
// low = EAX (bits 31:0), high = EDX (bits 63:32), msr in ECX
```

## 5. The SYSCALL/SYSRET Path

### 5.1 How SYSCALL Works (Hardware)

When user space executes the `syscall` instruction, the CPU does the following in a single instruction:

1. `rcx` ← `rip` (save return address)
2. `r11` ← `rflags` (save flags)
3. `rflags` ← `rflags & ~MSR_SYSCALL_MASK` (clear IF to disable interrupts, clear other bits)
4. `cs` ← `MSR_STAR[47:32]` (kernel code segment, `__KERNEL_CS`)
5. `ss` ← `MSR_STAR[47:32] + 8` (kernel data segment, `__KERNEL_DS`)
6. `rip` ← `MSR_LSTAR` (jump to `entry_SYSCALL_64`)

No stack switch happens — the CPU is now running at ring 0 with the old user-space RSP. The very first thing the entry code must do is switch to the kernel stack.

### 5.2 Kernel Entry Code

`entry_SYSCALL_64` (`arch/x86/entry/entry_64.S:88`):

```asm
SYM_CODE_START(entry_SYSCALL_64)
    swapgs                          ; Switch GS base from user to kernel
                                    ; (gives access to per-CPU data)

    movq    %rsp, PER_CPU_VAR(cpu_tss_rw + TSS_sp2)  ; Save user RSP
    movq    PER_CPU_VAR(pcpu_hot + X86_top_of_stack), %rsp  ; Load kernel RSP

    ; Build a pt_regs frame on the kernel stack:
    pushq   $__USER_DS              ; pt_regs->ss
    pushq   PER_CPU_VAR(cpu_tss_rw + TSS_sp2)  ; pt_regs->sp (user RSP)
    pushq   %r11                    ; pt_regs->flags
    pushq   $__USER_CS              ; pt_regs->cs
    pushq   %rcx                    ; pt_regs->ip (return address)
    pushq   %rax                    ; pt_regs->orig_ax (syscall number)

    PUSH_AND_CLEAR_REGS             ; Push all other registers, clear them
                                    ; (clearing prevents info leaks)

    movq    %rsp, %rdi              ; 1st arg to C handler = pointer to pt_regs
    movq    %rax, %rsi              ; 2nd arg = syscall number

    call    do_syscall_64           ; Call the C handler
```

### 5.3 SYSRET — Returning to User Space

`sysretq` is the reverse:

1. `rip` ← `rcx` (restore return address)
2. `rflags` ← `r11` (restore flags)
3. `cs` ← `MSR_STAR[63:48] + 16` (user code segment, `__USER_CS`)
4. `ss` ← `MSR_STAR[63:48] + 8` (user data segment, `__USER_DS`)
5. Switch to ring 3

The kernel restores registers from `pt_regs`, loads `rcx` and `r11` from the saved values, does `swapgs` to restore the user GS base, and executes `sysretq`.

### 5.4 Why swapgs?

The `swapgs` instruction swaps the value of `MSR_GS_BASE` with `MSR_KERNEL_GS_BASE`. In user mode, GS points to user-space thread-local storage. In kernel mode, GS must point to per-CPU data (so the kernel can find the current task, the kernel stack, etc.). `swapgs` switches between these two bases atomically.

## 6. Segmentation in Long Mode

### 6.1 Segments Are (Mostly) Flat

In 64-bit long mode, segmentation is largely vestigial. All segment bases are treated as 0 and limits as 2^64, with two exceptions:

- **CS** (Code Segment) — determines the privilege level (ring 0 vs ring 3) and the code mode (64-bit vs 32-bit compatibility)
- **FS and GS** — can have nonzero bases, set via MSRs. GS is used for per-CPU data (kernel) and thread-local storage (user space). FS is used for user-space thread-local storage (glibc sets it via `arch_prctl(ARCH_SET_FS, addr)`).

### 6.2 The GDT Layout

The Global Descriptor Table (`arch/x86/include/asm/segment.h:169–218`):

```
Entry  Index  Selector  Name                 DPL  Purpose
──────────────────────────────────────────────────────────────
  0      0     0x00     NULL                  —   Required null entry
  1      1     0x08     KERNEL32_CS           0   32-bit kernel code
  2      2     0x10     KERNEL_CS (__KERNEL_CS) 0  64-bit kernel code
  3      3     0x18     KERNEL_DS (__KERNEL_DS) 0  Kernel data
  4      4     0x20     USER32_CS             3   32-bit user code (compat)
  5      5     0x2b     USER_DS (__USER_DS)   3   User data (note: +3 for RPL)
  6      6     0x33     USER_CS (__USER_CS)   3   64-bit user code (note: +3)
  7      7     —        unused                —
  8-9    8     0x40     TSS                   0   Task State Segment (16 bytes)
  10-11  10    0x50     LDT                   0   Local Descriptor Table
  12-14  12-14 —        TLS                   3   Thread-Local Storage (3 entries)
  15     15    —        CPUNODE               3   CPU/NUMA node info
```

The RPL (Requested Privilege Level) in the bottom 2 bits of a selector determines the privilege level. `__KERNEL_CS = 0x10` (RPL=0, ring 0), `__USER_CS = 0x33` (RPL=3, ring 3).

The GDT layout is specifically arranged to satisfy the SYSRET instruction's hardcoded selector arithmetic — see the comment at `segment.h:177–186`.

### 6.3 The TSS (Task State Segment)

In long mode, the TSS is not used for hardware task switching (that is a 32-bit feature). Instead, it serves two purposes:

1. **RSP0**: When the CPU transitions from ring 3 to ring 0 (syscall, interrupt), it loads the kernel stack pointer from `TSS.RSP0`. This is how the CPU knows where the kernel stack is.

2. **IST (Interrupt Stack Table)**: The TSS contains 7 IST entries — alternative stack pointers for specific exceptions. When an exception uses an IST entry, the CPU switches to that stack unconditionally, even if already in kernel mode. This is critical for double faults, NMIs, and machine checks — if the kernel stack itself is corrupted, these handlers need a known-good stack.

IST indices (`arch/x86/include/asm/page_64_types.h:27–31`):

| Index | Exception | Why it needs its own stack |
|-------|-----------|---------------------------|
| 0 | Double Fault (#DF) | The kernel stack may be the cause |
| 1 | NMI | Can arrive at any time, even during kernel stack setup |
| 2 | Debug (#DB) | Needs isolation from debugged code |
| 3 | Machine Check (#MC) | Hardware error, cannot trust anything |
| 4 | VMM Communication (#VC) | SEV-ES secure VM communication |

## 7. Paging — Virtual to Physical Address Translation

### 7.1 The 4-Level Page Table

x86_64 uses a 4-level (or 5-level) hierarchical page table. A 48-bit virtual address is split into indices that walk through four levels of tables:

```
 63    48 47    39 38    30 29    21 20    12 11        0
┌────────┬────────┬────────┬────────┬────────┬──────────┐
│sign ext│  PGD   │  PUD   │  PMD   │  PTE   │  Offset  │
│(16 bits)│(9 bits)│(9 bits)│(9 bits)│(9 bits)│ (12 bits)│
└────────┴────────┴────────┴────────┴────────┴──────────┘
           │         │         │         │         │
           ▼         ▼         ▼         ▼         ▼
         CR3 ──► PGD ──► PUD ──► PMD ──► PTE ──► Physical Page
       (512 entries)(512)  (512)  (512)     + offset = physical addr
```

Each table has 512 entries (9 bits of index), each entry is 8 bytes, so each table is exactly one 4 KB page.

The constants from `arch/x86/include/asm/pgtable_64_types.h`:

```c
#define PAGE_SHIFT      12              // 4 KB pages
#define PMD_SHIFT       21              // 2 MB (huge page level)
#define PUD_SHIFT       30              // 1 GB (huge page level)
#define P4D_SHIFT       39              // 512 GB (5-level paging only)
#define PGDIR_SHIFT     39 or 48        // 4-level: 39, 5-level: 48

#define PTRS_PER_PTE    512
#define PTRS_PER_PMD    512
#define PTRS_PER_PUD    512
#define PTRS_PER_PGD    512
```

### 7.2 Page Table Entry Flags

Each page table entry is a 64-bit value. The physical page frame number occupies bits [51:12], and the remaining bits are flags (`arch/x86/include/asm/pgtable_types.h:10–30`):

| Bit | Name | Purpose |
|-----|------|---------|
| 0 | Present | Page is mapped. If 0, any access causes a page fault. |
| 1 | R/W | If 0, the page is read-only (for both user and kernel if CR0.WP=1). |
| 2 | U/S | If 0, only supervisor (ring 0) can access. This is the kernel/user split. |
| 3 | PWT | Page Write-Through (cache policy). |
| 4 | PCD | Page Cache Disable. |
| 5 | Accessed | Set by CPU when the page is read. Used by the page reclaim algorithm. |
| 6 | Dirty | Set by CPU when the page is written. Used by writeback. |
| 7 | PSE | Page Size Extension — if set in PMD, this is a 2 MB huge page; if set in PUD, a 1 GB huge page. No further table walk needed. |
| 8 | Global | TLB entry is not flushed on CR3 switch. Used for kernel mappings. |
| 63 | NX | No Execute — if set, the page cannot be executed. Requires EFER.NXE=1. Critical for W^X security (pages are writable XOR executable). |

### 7.3 Huge Pages

Instead of walking all 4 levels down to a 4 KB page, the kernel can set the PSE bit at the PMD level (creating a 2 MB huge page) or at the PUD level (creating a 1 GB huge page). The remaining address bits become the offset within the huge page. Huge pages reduce TLB pressure because one TLB entry covers more memory.

The kernel's direct mapping of physical memory (`page_offset_base`, typically starting at `0xffff888000000000`) often uses 1 GB huge pages for efficiency.

### 7.4 Canonical Addresses

x86_64 with 4-level paging only uses 48 bits of virtual address. Bits [63:48] must be a sign extension of bit 47 — either all 0s (user space) or all 1s (kernel space). Addresses that violate this rule are "non-canonical" and cause a General Protection Fault (#GP) if used.

This creates the address space split:

```
0x0000000000000000 – 0x00007FFFFFFFFFFF   User space   (~128 TB)
0x0000800000000000 – 0xFFFF7FFFFFFFFFFF   Non-canonical hole
0xFFFF800000000000 – 0xFFFFFFFFFFFFFFFF   Kernel space (~128 TB)
```

With 5-level paging (57 bits, `X86_FEATURE_LA57`), user space grows to ~64 PB.

### 7.5 The Virtual Address Space Layout

From `Documentation/arch/x86/x86_64/mm.rst` (4-level paging):

| Start Address | Size | Region |
|--------------|------|--------|
| `0x0000000000000000` | ~128 TB | User-space memory (per-process) |
| `0xffff800000000000` | 8 TB | Guard hole / hypervisor |
| `0xffff880000000000` | 0.5 TB | LDT remap for PTI (page table isolation) |
| `0xffff888000000000` | 64 TB | **Direct mapping** of all physical memory |
| `0xffffc90000000000` | 32 TB | **vmalloc / ioremap** space |
| `0xffffea0000000000` | 1 TB | **vmemmap** (struct page array) |
| `0xffffec0000000000` | 16 TB | KASAN shadow memory |
| `0xfffffe0000000000` | 0.5 TB | cpu_entry_area (per-CPU, always mapped) |
| `0xffffffff80000000` | 512 MB | **Kernel text** (mapped to physical 0) |
| `0xffffffffa0000000` | 1520 MB | **Module mapping** space |

Key observations:
- The **direct mapping** (`page_offset_base`) maps all physical RAM into the kernel's virtual address space. If you have 64 GB of RAM, the first 64 GB of this 64 TB region is populated. `__va(phys)` and `__pa(virt)` convert between physical and virtual addresses in this region.
- **vmalloc** space is for virtually contiguous but physically scattered allocations.
- **vmemmap** is the virtual memory map — it holds the `struct page` array that describes every physical page frame.
- The kernel text region at `-2 GB` is where the kernel image itself lives. Module code is loaded just above it.

### 7.6 KASLR (Kernel Address Space Layout Randomization)

When `CONFIG_RANDOMIZE_MEMORY` is enabled, the bases of the direct mapping, vmalloc, and vmemmap regions are randomized at boot. This makes it harder for exploits to predict kernel addresses. The kernel text base is also randomized by `CONFIG_RANDOMIZE_BASE`.

## 8. The Interrupt Descriptor Table (IDT)

### 8.1 What the IDT Does

The IDT maps exception/interrupt vectors (0–255) to handler functions. When the CPU encounters an exception (e.g., page fault, vector 14) or receives a hardware interrupt (e.g., vector 33), it looks up the handler in the IDT and jumps to it.

### 8.2 IDT Entry Structure

`struct gate_struct` (`arch/x86/include/asm/desc_defs.h:119–143`):

```c
struct idt_bits {
    u16 ist   : 3,      // IST index (0 = use RSP0 from TSS, 1-7 = IST stack)
       zero  : 5,
       type  : 5,       // GATE_INTERRUPT (0xE) or GATE_TRAP (0xF)
       dpl   : 2,       // Descriptor Privilege Level (0=kernel, 3=user)
       p     : 1;       // Present
};

struct gate_struct {
    u16           offset_low;    // Handler address bits [15:0]
    u16           segment;       // Code segment selector (__KERNEL_CS)
    struct idt_bits bits;
    u16           offset_middle; // Handler address bits [31:16]
    u32           offset_high;   // Handler address bits [63:32]
    u32           reserved;
};
```

### 8.3 IDT Setup in the Kernel

The IDT is populated in `arch/x86/kernel/idt.c:84–120`:

```c
static const __initconst struct idt_data def_idts[] = {
    INTG(X86_TRAP_DE,   asm_exc_divide_error),         // #0  Divide Error
    ISTG(X86_TRAP_NMI,  asm_exc_nmi, IST_INDEX_NMI),   // #2  NMI (uses IST)
    INTG(X86_TRAP_BR,   asm_exc_bounds),                // #5  Bound Range Exceeded
    INTG(X86_TRAP_UD,   asm_exc_invalid_op),            // #6  Invalid Opcode
    INTG(X86_TRAP_NM,   asm_exc_device_not_available),  // #7  Device Not Available
    INTG(X86_TRAP_TS,   asm_exc_invalid_tss),           // #10 Invalid TSS
    INTG(X86_TRAP_NP,   asm_exc_segment_not_present),   // #11 Segment Not Present
    INTG(X86_TRAP_SS,   asm_exc_stack_segment),         // #12 Stack-Segment Fault
    INTG(X86_TRAP_GP,   asm_exc_general_protection),    // #13 General Protection
    INTG(X86_TRAP_MF,   asm_exc_coprocessor_error),     // #16 x87 FPU Error
    INTG(X86_TRAP_AC,   asm_exc_alignment_check),       // #17 Alignment Check
    INTG(X86_TRAP_XF,   asm_exc_simd_coprocessor_error),// #19 SIMD FP Exception
    ISTG(X86_TRAP_DF,   asm_exc_double_fault, IST_INDEX_DF),  // #8 Double Fault
    ISTG(X86_TRAP_DB,   asm_exc_debug, IST_INDEX_DB),         // #1 Debug
    ISTG(X86_TRAP_MC,   asm_exc_machine_check, IST_INDEX_MCE),// #18 Machine Check
    SYSG(X86_TRAP_OF,   asm_exc_overflow),              // #4  Overflow (DPL=3)
};
```

The macros:
- **`INTG`** — Interrupt Gate, DPL=0 (only kernel can trigger via INT instruction)
- **`ISTG`** — Interrupt Gate with IST stack (for exceptions that need their own stack)
- **`SYSG`** — System Gate, DPL=3 (user space can trigger, e.g., `int3` for breakpoints)

### 8.4 What the CPU Pushes on an Exception

When an exception occurs, the CPU automatically pushes (onto the kernel stack or IST stack):

```
High address (pushed first)
┌──────────────────────┐
│        SS            │  ← Only if privilege change (ring 3 → ring 0)
│        RSP           │  ← Only if privilege change
│       RFLAGS         │
│        CS            │
│        RIP           │  ← Return address (where the fault happened)
│    Error Code        │  ← Only for some exceptions (#PF, #GP, #DF, etc.)
└──────────────────────┘
Low address (stack top after push)
```

Not all exceptions push an error code. The kernel must know which do and which do not — this is why there are separate `DEFINE_IDTENTRY` and `DEFINE_IDTENTRY_ERRORCODE` macros in `idtentry.h`.

### 8.5 Interrupt Gate vs Trap Gate

Both point to a handler, but they differ in one critical way:

- **Interrupt Gate** (type 0xE): The CPU **clears IF** (disables interrupts) before jumping to the handler. Used for all hardware interrupts and most exceptions.
- **Trap Gate** (type 0xF): The CPU **does not change IF**. Used for debugging exceptions where you want interrupts to remain as they were.

## 9. Inline Assembly in the Kernel

### 9.1 The Basic Syntax

The kernel uses GCC extended inline assembly extensively:

```c
asm volatile("instruction"
    : output_operands
    : input_operands
    : clobbers);
```

- **`volatile`** — do not optimize away or reorder this assembly
- **Output operands** — `"=r"(var)` means "put result in a register, assign to `var`"
- **Input operands** — `"r"(var)` means "put `var` in a register"
- **Clobbers** — `"memory"` means this instruction affects memory (prevents compiler reordering across it), `"cc"` means it affects condition codes

### 9.2 Common Constraint Letters

| Constraint | Meaning |
|-----------|---------|
| `"r"` | Any general-purpose register |
| `"m"` | Memory operand |
| `"i"` | Immediate (compile-time constant) |
| `"a"` | `rax`/`eax`/`ax`/`al` |
| `"b"` | `rbx` |
| `"c"` | `rcx` (used for `ecx` in MSR operations) |
| `"d"` | `rdx` |
| `"q"` | `rax`, `rbx`, `rcx`, or `rdx` (byte-addressable) |
| `"=r"` | Output in register (write-only) |
| `"+r"` | Input/output in register (read-modify-write) |
| `"0"` | Same register as operand 0 (ties input to output) |
| `"er"` | Register or sign-extended 32-bit immediate |

### 9.3 Example: Reading CR3

```c
static inline unsigned long __native_read_cr3(void)
{
    unsigned long val;
    asm volatile("mov %%cr3, %0" : "=r" (val));
    //                             ^^^^^^^^^^^
    //                             Output: put CR3 value in 'val'
    //                             "=r" = any register, write-only
    return val;
}
```

### 9.4 Example: Atomic Compare-and-Exchange

From `arch/x86/include/asm/cmpxchg.h:78–156`:

```c
// Simplified from the actual macro expansion:
asm_inline volatile(LOCK_PREFIX "cmpxchgq %2, %1"
    : "=a" (__ret), "+m" (*__ptr)     // outputs: rax = old value, memory updated
    : "r" (__new), "0" (__old)        // inputs: new value in reg, old value in rax
    : "memory");                      // clobber: affects memory
```

This compiles to `lock cmpxchg [ptr], new_val`:
- The CPU compares `rax` (old value) with `[ptr]`
- If equal: `[ptr] ← new_val`, ZF=1
- If not equal: `rax ← [ptr]`, ZF=0
- The `lock` prefix makes the entire operation atomic

### 9.5 Example: Atomic Add

From `arch/x86/include/asm/atomic64_64.h:23`:

```c
static __always_inline void arch_atomic64_add(s64 i, atomic64_t *v)
{
    asm_inline volatile(LOCK_PREFIX "addq %1, %0"
        : "=m" (v->counter)           // output: memory location
        : "er" (i), "m" (v->counter)  // inputs: value and current memory
        : "memory");
}
```

The `lock addq` instruction atomically adds `i` to `v->counter`. No load-modify-store race is possible because `lock` holds the cache line exclusively for the entire operation.

### 9.6 Example: Bit Test and Set

From `arch/x86/include/asm/bitops.h:51`:

```c
static __always_inline void arch_set_bit(long nr, volatile unsigned long *addr)
{
    if (__builtin_constant_p(nr)) {
        asm_inline volatile(LOCK_PREFIX "orb %b1, %0"
            : CONST_MASK_ADDR(nr, addr)
            : "iq" (CONST_MASK(nr))
            : "memory");
    } else {
        asm_inline volatile(LOCK_PREFIX __ASM_SIZE(bts) " %1, %0"
            : : RLONG_ADDR(addr), "Ir" (nr) : "memory");
    }
}
```

For constant bit numbers, this uses `lock or` (byte operation). For variable bit numbers, it uses `lock bts` (Bit Test and Set) — an x86 instruction specifically designed for this.

## 10. Memory Barriers and Ordering

### 10.1 Why Barriers Are Needed

x86_64 has a relatively strong memory model (Total Store Order), but it is not sequentially consistent. The CPU can:
- Reorder loads past earlier stores (to different addresses)
- Merge stores to the same address
- The *compiler* can reorder anything it wants unless told otherwise

The kernel uses barriers to prevent unwanted reordering.

### 10.2 Hardware Barriers

| Instruction | Purpose | Kernel Wrapper |
|------------|---------|----------------|
| `mfence` | Full fence — no loads or stores cross this point | `mb()` |
| `lfence` | Load fence — no later load passes an earlier load | `rmb()` |
| `sfence` | Store fence — no later store passes an earlier store | `wmb()` |
| `lock` prefix | Full barrier (side effect of the atomic operation) | — |

On x86_64, stores are already ordered with respect to other stores, and loads with respect to other loads. So `wmb()` and `rmb()` compile to compiler barriers only (no actual `sfence`/`lfence` instruction) — the hardware already provides that ordering. Only `mb()` (full barrier, needed when loads must be ordered with respect to stores) compiles to `mfence` or a `lock` instruction.

### 10.3 Compiler Barriers

```c
barrier();    // asm volatile("" ::: "memory")
```

This generates no machine code — it only tells the compiler "do not reorder memory accesses across this point." Used when the hardware already provides the needed ordering, but the compiler might reorder things.

### 10.4 READ_ONCE / WRITE_ONCE

```c
int val = READ_ONCE(shared_var);
WRITE_ONCE(shared_var, new_val);
```

These prevent the compiler from:
- Splitting a single access into multiple accesses
- Merging multiple accesses into one
- Caching the value in a register across a barrier

They compile to a `volatile` access — one load or one store, guaranteed. Essential for any variable shared between threads or between a thread and an interrupt handler.

## 11. CPU Feature Detection and Alternatives

### 11.1 CPU Feature Flags

The kernel detects CPU features at boot using the `cpuid` instruction and stores the results in `boot_cpu_data.x86_capability[]`. Feature constants are defined in `arch/x86/include/asm/cpufeatures.h`:

```c
/* CPUID leaf 1, EDX */
#define X86_FEATURE_FPU         (0*32 + 0)   // x87 FPU
#define X86_FEATURE_TSC         (0*32 + 4)   // Time Stamp Counter
#define X86_FEATURE_MSR         (0*32 + 5)   // Model-Specific Registers
#define X86_FEATURE_PAE         (0*32 + 6)   // Physical Address Extensions
#define X86_FEATURE_APIC        (0*32 + 9)   // Local APIC

/* CPUID leaf 0x80000001, EDX (AMD) */
#define X86_FEATURE_SYSCALL     (1*32 + 11)  // SYSCALL/SYSRET
#define X86_FEATURE_NX          (1*32 + 20)  // No-Execute bit
#define X86_FEATURE_LM          (1*32 + 29)  // Long Mode (64-bit)

/* Kernel-defined features */
#define X86_FEATURE_CONSTANT_TSC (3*32 + 8)  // TSC ticks at constant rate
#define X86_FEATURE_REP_GOOD    (3*32 + 16)  // REP prefix is fast
#define X86_FEATURE_NOPL        (3*32 + 20)  // NOPL instruction
```

### 11.2 Testing Features

From `arch/x86/include/asm/cpufeature.h:51–89`:

```c
// Runtime check (reads boot_cpu_data):
if (boot_cpu_has(X86_FEATURE_NX))
    enable_nx();

// Static key optimization (patches code at boot):
if (cpu_feature_enabled(X86_FEATURE_SMEP))
    // This branch is patched at boot time to either always-taken or never-taken
```

### 11.3 The Alternatives Mechanism

The kernel uses a sophisticated runtime patching system (`arch/x86/include/asm/alternative.h`) to optimize code paths based on CPU features. At boot time, after feature detection, the kernel walks through all `alternative()` sites and patches the code in place:

```c
// From special_insns.h — CLFLUSHOPT with fallback:
static inline void clflushopt(volatile void *__p)
{
    alternative_io("ds clflush %0",        // default: plain CLFLUSH
                   "clflushopt %0",         // replacement: CLFLUSHOPT
                   X86_FEATURE_CLFLUSHOPT,  // use replacement if feature present
                   "+m" (*(volatile char *)__p));
}
```

If the CPU supports `CLFLUSHOPT`, the kernel patches the `ds clflush` instruction to `clflushopt` at boot time. Otherwise, it leaves the default. This avoids a runtime branch on every call.

The alternatives mechanism is also used for:
- Spectre mitigations (patching in `lfence` or `retpoline` calls)
- Lock prefix optimization (removing `lock` on UP systems)
- NOP padding (replacing NOPs with the CPU's preferred NOP sequence)

## 12. The TLB (Translation Lookaside Buffer)

### 12.1 What the TLB Does

The TLB caches virtual-to-physical address translations. Without the TLB, every memory access would require a 4-level page table walk (4 memory reads). The TLB stores recent translations so the CPU can skip the walk.

### 12.2 TLB Flushing

When the kernel modifies page tables, it must invalidate the affected TLB entries. The key functions are in `arch/x86/include/asm/tlbflush.h` and `arch/x86/mm/tlb.c`.

**Single-page flush:**

```c
// Invalidate TLB for one virtual address:
static inline void __native_flush_tlb_one_user(unsigned long addr)
{
    asm volatile("invlpg (%0)" ::"r" (addr) : "memory");
}
```

The `invlpg` instruction invalidates the TLB entry for a single virtual address. Used when a single page's mapping changes (e.g., `mprotect()` on one page).

**Full TLB flush:**

Writing to CR3 flushes all non-global TLB entries. The kernel does this on context switch (when changing address spaces). With PCID support, the kernel can write CR3 with the NOFLUSH bit set to avoid flushing entries tagged with the new PCID.

### 12.3 TLB Shootdown — The Multi-CPU Problem

When one CPU modifies a page table, the TLB entries on *other* CPUs are still stale. The kernel must send an IPI (Inter-Processor Interrupt) to those CPUs, asking them to flush their TLBs. This is called a **TLB shootdown**.

The mechanism (`arch/x86/mm/tlb.c`):

1. CPU 0 modifies a page table entry
2. CPU 0 calls `flush_tlb_mm_range()` → `flush_tlb_multi()`
3. This sends an IPI to all CPUs that have the affected mm loaded
4. Each receiving CPU runs `flush_tlb_func()` in the IPI handler
5. The handler executes `invlpg` for each affected address (or a full CR3 reload if many addresses changed)
6. CPU 0 waits for all IPIs to complete before returning

TLB shootdowns are expensive — they involve cross-CPU interrupts and synchronization. This is one reason why kernel developers are careful about page table modifications, and why lazy TLB (not shooting down when a CPU is running kernel code for a different mm) is an important optimization.

### 12.4 PCID/ASID — Avoiding TLB Flushes

PCID (Process Context Identifier) tags TLB entries with a 12-bit ID. When the kernel switches from process A to process B, it loads CR3 with process B's page table and a different PCID. TLB entries for process A are not flushed — they remain tagged with A's PCID. If the kernel switches back to A, those entries are still valid.

The kernel manages PCIDs per-CPU (`arch/x86/mm/tlb.c:64–91`):
- Each CPU has `TLB_NR_DYN_ASIDS` (typically 6) dynamically assigned ASIDs
- When an mm is scheduled on a CPU, it gets an ASID
- If all ASIDs are in use, the oldest one is evicted (and its TLB entries flushed)
- With KPTI, each mm uses two PCIDs: one for kernel mode (kPCID) and one for user mode (uPCID = kPCID + 2048)

## 14. KPTI — Kernel Page Table Isolation

### 14.1 The Meltdown Problem

The Meltdown vulnerability (2018) allowed user-space code to read kernel memory by exploiting speculative execution. The CPU would speculatively load kernel data into the cache before the permission check retired, and a side-channel attack could extract the data from the cache.

### 14.2 The Solution: Separate Page Tables

KPTI (Kernel Page Table Isolation) maintains two sets of page tables per process:

- **Kernel page tables**: Map everything — kernel text, kernel data, direct mapping, and user space
- **User page tables**: Map only user space and a minimal kernel "trampoline" (the entry/exit code and the IDT)

When running in user mode, CR3 points to the user page tables — the kernel is not mapped at all, so Meltdown cannot leak kernel data. On syscall/interrupt entry, the kernel switches CR3 to the full page tables.

### 14.3 Performance Impact

KPTI adds cost to every kernel entry/exit (CR3 write, TLB flush). With PCID support, this cost is reduced — the kernel and user page tables use different PCIDs, so switching between them does not flush the entire TLB.

The `cpu_entry_area` mapping at `0xfffffe0000000000` is the minimal kernel mapping present in user page tables — it contains the entry trampolines, the IDT, and the per-CPU TSS (so the CPU can find the kernel stack on entry).

## 15. Per-CPU Data and the GS Register

### 15.1 The Problem: Shared Data on SMP

On a multiprocessor system, global variables are shared between all CPUs. Accessing them requires locking, which is slow. Many kernel subsystems need per-CPU counters, caches, and state — data that each CPU accesses frequently but that does not need to be shared.

### 15.2 How Per-CPU Variables Work on x86_64

The kernel uses the GS segment register to access per-CPU data without any locking:

1. At boot, the kernel allocates a separate copy of all per-CPU variables for each CPU
2. Each CPU's GS base (`MSR_GS_BASE`) is set to point to that CPU's per-CPU area
3. Per-CPU variable access uses `%gs:offset` addressing — the offset is the same on all CPUs, but the GS base is different, so each CPU reads its own copy

```c
// Accessing a per-CPU variable (simplified):
#define this_cpu_read(var)    __this_cpu_read(var)

// Compiles to something like:
// movq %gs:offset_of_var, %rax
```

### 15.3 Per-CPU Data in the Kernel Entry Path

The first thing the kernel does on syscall entry is `swapgs` — this sets GS to point to the current CPU's per-CPU area. Then it can immediately access:

- `cpu_current_top_of_stack` — the kernel stack pointer for the current task
- `current_task` — pointer to the current `struct task_struct`
- `cpu_number` — which CPU we are on
- `softirq_pending` — pending softirqs for this CPU
- IRQ stacks, NMI stacks, etc.

```c
// The kernel's "current" macro on x86_64:
DECLARE_PER_CPU(struct task_struct *, current_task);
#define current  this_cpu_read_stable(current_task)
// Compiles to: movq %gs:current_task_offset, %rax
```

This is why `current` is so fast on x86_64 — it is a single `mov` instruction with a GS-relative address, no function call, no locking, no memory indirection beyond the one load.

### 15.4 The pcpu_hot Structure

Frequently accessed per-CPU data is grouped into `struct pcpu_hot` to share a cache line:

```c
struct pcpu_hot {
    union {
        struct {
            struct task_struct  *current_task;       // current process
            int                 preempt_count;       // preemption/softirq nesting
            int                 cpu_number;          // this CPU's ID
            unsigned long       top_of_stack;        // kernel stack top
            void                *hardirq_stack_ptr;  // hardirq stack
            u16                 softirq_pending;     // pending softirqs
            bool                preempt_lazy;        // lazy preemption
        };
        u8  pad[64];  // pad to one cache line (64 bytes)
    };
};
```
